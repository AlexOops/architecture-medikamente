# Меры и инструменты защиты данных

## Общие меры

| Мера | Где применять |
|---|---|
| RBAC | роли: врач, ресепшен, кассир, бухгалтер, склад, администратор |
| ABAC | доступ по филиалу, пациенту, врачу, цели доступа |
| MFA | доступ сотрудников к системам с PII и MEDICAL |
| Audit Log | чтение, изменение, экспорт, удаление данных |
| Encryption at rest | БД, файловое хранилище, резервные копии |
| Encryption in transit | API, интеграции, админские подключения |
| DLP | почта, файловый сервер, выгрузки |
| Data Masking | UI, отчёты, логи |
| Anonymization | BI/ML/AI, тестовые среды |
| Retention Policy | сроки хранения и удаление данных |
| Data Lineage | ETL, BI, ML, API-интеграции |
| Data Catalog | владельцы, теги, источники, схемы |
| API Gateway | внешние интеграции и лаборатория |
| SIEM | мониторинг доступа и алерты |

## Процесс регистрации пациента

- минимизировать состав собираемых данных;
- разделить обязательные и расширенные поля;
- хранить согласия отдельно;
- шифровать документы и сканы;
- логировать доступ к карточке пациента.

## Процесс приёма врача

- доступ врача только к своим пациентам;
- медицинские записи хранить в защищённой БД;
- запретить хранение диагнозов в Excel;
- включить audit log на чтение и изменение медкарты.

## Процесс лабораторных анализов

- интеграция только через API Gateway;
- передавать лаборатории только минимальный набор данных;
- использовать pseudonymous patient_id;
- результаты анализов тегировать как MEDICAL;
- логировать получение результатов.

## Процесс оплаты

- разделить медицинские данные и платёжные данные;
- кассиру не нужен доступ к диагнозам;
- в 1С передавать только финансовые реквизиты;
- фискальные данные хранить отдельно от medical data.

## BI/ML/AI

- использовать обезличенные или псевдонимизированные данные;
- запретить прямые Excel-выгрузки с PII;
- фиксировать lineage от источника до витрины;
- проверять датасеты на наличие запрещённых тегов.
