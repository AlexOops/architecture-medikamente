# Проблемные зоны As-Is

## 1. Хранение данных в Excel, PDF, JPG и сканах

Проблема:

- данные пациентов хранятся на общем файловом сервере;
- формат хранения неуправляемый;
- нет единой модели данных;
- сложно контролировать доступ;
- сложно удалять данные по запросу пациента.

Риск:

- утечка PII и медицинских данных;
- потеря целостности данных;
- невозможность доказать, кто и когда получил доступ.

Что улучшить:

- перенести критичные данные в управляемые БД;
- оставить файлы только как вложения с метаданными;
- включить шифрование, DLP и аудит доступа.

## 2. Нет классификации данных

Проблема:

- система не различает обычные данные, PII, медицинские данные, платёжные данные;
- сотрудники могут работать с файлами без понимания уровня чувствительности.

Риск:

- избыточный доступ;
- неправильная передача данных во внешние системы;
- нарушение принципа Data Minimization.

Что улучшить:

- ввести data tags;
- классифицировать данные по уровню чувствительности;
- использовать теги в API, DWH, BI и логировании.

## 3. Нет RBAC/ABAC

Проблема:

- доступ ограничивается в основном доменной аутентификацией;
- нет детального разграничения по ролям, филиалам, врачу, пациенту и назначению доступа.

Риск:

- врач может получить лишние данные;
- сотрудник ресепшена может увидеть медицинские сведения;
- пациент в будущем может получить чужие данные через ЛК.

Что улучшить:

- RBAC для базовых ролей;
- ABAC для филиала, врача, пациента, типа данных и цели доступа.

## 4. Нет аудита действий

Проблема:

- нет журналирования чтения, изменения, выгрузки и удаления медицинских данных;
- нет централизованного мониторинга.

Риск:

- невозможно расследовать инцидент;
- невозможно доказать легитимность доступа;
- нет алертов на нетипичные действия.

Что улучшить:

- audit log для всех операций с PII и medical data;
- отправка событий в SIEM;
- алерты на массовую выгрузку и доступ вне роли.

## 5. Неконтролируемые внутренние потоки данных

Проблема:

- Excel, 1С, файловый сервер, почта и лаборатория обмениваются данными без единой политики;
- данные могут уходить во вложениях, выгрузках и сканах.

Риск:

- утечка через почту;
- копирование файлов на рабочие станции;
- передача лишних данных партнёрам.

Что улучшить:

- API Gateway для внешних интеграций;
- DLP на почту и файловый сервер;
- data contracts;
- минимизация состава данных в API.

## 6. Файловый режим 1С

Проблема:

- 1С работает в файловом режиме;
- сложнее контролировать доступ, резервное копирование и аудит.

Риск:

- повреждение данных;
- слабая масштабируемость при росте филиалов;
- сложность централизованного контроля.

Что улучшить:

- перевести 1С на серверный режим;
- настроить разграничение доступа;
- включить резервное копирование и аудит.

## 7. Нет Data Lineage

Проблема:

- непонятно, из каких источников сформированы отчёты;
- нет трассировки движения данных из Excel/CRM/лаборатории в BI/ML/AI.

Риск:

- сложно управлять качеством данных;
- сложно выполнять удаление/обезличивание;
- сложно расследовать утечки.

Что улучшить:

- ввести каталог данных;
- фиксировать lineage для ETL/ELT;
- хранить владельца, источник, теги и срок хранения данных.
