# Стратегия защиты данных

## 1. Защита данных при хранении

### Базы данных

Применять:

- Transparent Data Encryption или шифрование дисков;
- шифрование отдельных чувствительных полей;
- управление ключами через KMS/Vault;
- RBAC/ABAC на уровне приложений;
- ограничение прямого доступа к БД;
- audit log для чтения и изменения PII/MEDICAL данных.

Где применять:

- Patient DB;
- Medical DB;
- Payment DB;
- Document Storage;
- Data Lake;
- DWH;
- backup-хранилище.

### Файлы и документы

Применять:

- encrypted object storage;
- версионирование;
- запрет хранения медицинских карт в общих папках;
- контроль скачивания;
- DLP;
- retention policy.

Файловый сервер должен перестать быть основной системой учёта медицинских данных.

### Резервные копии

Применять:

- шифрование backup;
- immutable backup;
- отдельные права на restore;
- регулярную проверку восстановления;
- журналирование операций восстановления.

## 2. Защита данных при передаче

### Внутренние API

Применять:

- TLS 1.2+;
- mTLS между критичными сервисами;
- service-to-service authentication;
- API Gateway;
- schema validation;
- rate limiting;
- request/response masking;
- запрет PII/MEDICAL в URL query parameters.

### Внешние интеграции

Для лаборатории и будущих партнёров:

- API Gateway;
- mTLS;
- OAuth2 Client Credentials;
- allowlist внешних систем;
- data contracts;
- минимизация состава данных;
- pseudonymous patient_id вместо ФИО, где это возможно;
- audit log каждого запроса.

## 3. Защита данных при использовании

### Пользовательские интерфейсы

Применять:

- masking телефона, email, документов;
- доступ по роли и атрибутам;
- запрет показа medical data сотрудникам без необходимости;
- ограничение массовых выгрузок;
- водяные знаки для выгрузок;
- audit log просмотра карточки пациента.

### Backend-обработка

Применять:

- policy checks до выполнения операции;
- запрет логирования PII/MEDICAL значений;
- secrets только через Vault;
- разделение доменов PII, MEDICAL, PAYMENT;
- проверка ownership для ЛК пациента.

## 4. Защита аналитики, BI, ML и AI

Аналитический слой не должен работать с сырыми PII/MEDICAL данными по умолчанию.

Применять:

- ingestion только через контролируемый pipeline;
- data classification;
- data tagging;
- anonymization;
- pseudonymization;
- data lineage;
- data catalog;
- отдельные роли для аналитиков;
- отдельные sandbox-окружения;
- запрет выгрузки critical datasets без согласования.

Поток:

```text
Source Systems
-> Data Classification
-> Policy Check
-> Anonymization / Pseudonymization
-> Data Lake
-> DWH / Data Marts
-> BI / ML / AI
```

## 5. Управление ключами и секретами

Применять:

- Vault/KMS для ключей и секретов;
- запрет секретов в Git;
- rotation ключей;
- разные ключи для разных классов данных;
- ограничение доступа к ключам;
- аудит операций с ключами.

Для особо критичных ключей можно использовать HSM.

## 6. Удаление и прекращение обработки данных

Применять:

- retention policy;
- workflow на запрос пациента;
- удаление или обезличивание данных в производственных системах;
- удаление или обезличивание связанных данных в аналитике;
- lineage для поиска производных наборов данных;
- журналирование факта удаления.

## 7. Минимальный набор базовых инструментов

Первый этап внедрения:

1. IAM / SSO + MFA.
2. RBAC для сотрудников.
3. ABAC для medical data.
4. TLS для API.
5. Шифрование БД и backup.
6. Audit log для PII/MEDICAL.
7. DLP для файлов и почты.
8. SIEM для событий доступа.
9. Vault/KMS для секретов и ключей.
10. Data Catalog + тегирование данных.
