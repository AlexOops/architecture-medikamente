# Архитектурные решения To-Be

## 1. Единый IAM / SSO

Добавляется единый IAM-контур.

Назначение:

- единая аутентификация сотрудников и клиентов;
- MFA для сотрудников;
- централизованное управление ролями;
- поддержка RBAC и ABAC;
- разделение доступа по филиалу, роли, врачу, пациенту и цели доступа.

Примеры ролей:

- patient;
- doctor;
- reception;
- cashier;
- accountant;
- warehouse;
- admin;
- analyst;
- integration-service.

## 2. Policy Decision Point

Добавляется отдельный сервис проверки политик доступа.

Он принимает решение:

- можно ли пользователю получить данные;
- можно ли системе передать данные наружу;
- можно ли выгрузить данные в BI/ML/AI;
- требуется ли маскирование или обезличивание.

Факторы решения:

- роль пользователя;
- филиал;
- пациент;
- врач;
- тип данных;
- теги данных;
- цель доступа;
- правовое основание;
- согласие пациента.

## 3. Data Classification & Tagging

Все данные получают теги чувствительности.

Основные теги:

- PUBLIC;
- INTERNAL;
- PII;
- PII_EXTENDED;
- MEDICAL;
- PAYMENT;
- LEGAL;
- HR;
- ANONYMIZED;
- PSEUDONYMIZED.

Теги используются в:

- API Gateway;
- CRM;
- EMR;
- лабораторной интеграции;
- Data Lake;
- DWH;
- BI/ML/AI;
- аудитах и алертах.

## 4. Consent Management

Добавляется сервис управления согласиями пациента.

Он хранит:

- согласие на обработку персональных данных;
- согласие на медицинскую обработку;
- согласие на передачу данных партнёрам;
- отзыв согласия;
- дату и версию согласия.

Без валидного согласия данные нельзя использовать в новых процессах, если нет другого правового основания.

## 5. API Gateway для интеграций

Все внешние интеграции проходят через API Gateway.

Назначение:

- аутентификация внешних систем;
- авторизация запросов;
- rate limiting;
- schema validation;
- data minimization;
- masking;
- audit log.

Для лаборатории передаётся минимальный набор данных. Вместо ФИО используется pseudonymous patient_id, если полный состав PII не нужен.

## 6. Защищённые доменные системы

В целевом состоянии появляются управляемые доменные системы:

- Patient Portal;
- Staff Portal;
- CRM;
- EMR;
- Payment Gateway;
- Laboratory Integration Service.

Данные больше не должны храниться как основная система учёта в Excel, PDF, JPG и общих папках.

Файлы могут оставаться только как вложения в защищённом документном хранилище.

## 7. Audit Log

Все операции с PII и MEDICAL данными пишутся в audit log.

Фиксируются:

- кто получил доступ;
- к каким данным;
- с какой целью;
- когда;
- из какой системы;
- результат операции;
- факт выгрузки или просмотра.

Audit log должен быть защищён от изменения.

## 8. DLP и SIEM

DLP контролирует:

- файловый сервер;
- почту;
- выгрузки;
- вложения;
- массовое копирование.

SIEM собирает события:

- доступ к медицинским данным;
- массовые выгрузки;
- доступ вне роли;
- нетипичное время доступа;
- ошибки авторизации;
- попытки доступа к чужим данным.

## 9. Retention & Deletion

Добавляется механизм управления сроками хранения и удалением данных.

Назначение:

- удаление данных по окончании срока хранения;
- прекращение обработки по запросу пациента;
- удаление или обезличивание данных в аналитическом слое;
- контроль связанных копий данных через lineage.

## 10. Secure Data Platform

Для аналитики выделяется отдельный слой.

Компоненты:

- Data Ingestion;
- Data Classification;
- Anonymization / Pseudonymization;
- Data Lake;
- DWH / Data Marts;
- Data Catalog;
- Data Lineage;
- BI/ML/AI workspace.

Аналитики и ML/AI-сервисы работают не с сырыми медицинскими данными, а с подготовленными наборами данных.
