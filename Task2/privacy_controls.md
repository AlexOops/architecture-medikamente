# Privacy by Design механизмы

## Матрица механизмов

| Риск | Механизм | Где применяется |
|---|---|---|
| Избыточный сбор данных | Data Minimization | формы пациента, API, лаборатория |
| Доступ не по роли | RBAC / ABAC | CRM, EMR, порталы, отчёты |
| Доступ к чужим данным | Ownership check | ЛК пациента, мобильное приложение, API |
| Утечка через файлы | DLP | файловое хранилище, почта, выгрузки |
| Утечка через API | API Gateway | лаборатория, партнёры, клиентские сервисы |
| Передача лишних PII партнёрам | Data Contracts | внешние интеграции |
| Использование данных без согласия | Consent Management | CRM, EMR, аналитика |
| Невозможно расследовать инцидент | Audit Log + SIEM | все операции с PII/MEDICAL |
| Сырые данные в BI/ML/AI | Anonymization / Pseudonymization | Data Lake, DWH, ML datasets |
| Нет понимания происхождения данных | Data Lineage | ETL/ELT, витрины, отчёты |
| Нарушение сроков хранения | Retention & Deletion | CRM, EMR, документы, Data Lake |
| PII в логах | Masking / Log Filtering | backend, API Gateway, SIEM |

## Privacy by Design принципы

### 1. Proactive not reactive

Механизмы защиты проектируются до реализации новых сервисов:

- теги данных;
- политики доступа;
- audit log;
- data contracts;
- проверка релизов на работу с тегированными данными.

### 2. Privacy as default

По умолчанию доступ закрыт.

Доступ открывается только при наличии:

- роли;
- атрибута;
- цели доступа;
- правового основания;
- согласия пациента, если оно требуется.

### 3. Privacy embedded into design

Защита встроена в архитектуру:

- IAM;
- Policy Decision Point;
- API Gateway;
- Data Catalog;
- DLP;
- SIEM;
- Secure Data Platform.

### 4. Data Minimization

Системы получают только тот набор данных, который нужен для процесса.

Примеры:

- кассиру не нужен диагноз;
- лаборатории не нужен полный профиль пациента;
- BI/ML не нужны прямые идентификаторы пациента;
- пациент в ЛК видит только свои данные.

### 5. End-to-end security

Данные защищаются на всём пути:

```text
создание -> хранение -> передача -> аналитика -> архив -> удаление
```

### 6. Visibility and transparency

Должны быть доступны:

- audit log;
- data lineage;
- каталог данных;
- история согласий;
- события доступа.

### 7. Respect for user privacy

Пациент должен иметь возможность:

- видеть свои данные;
- управлять согласиями;
- запрашивать прекращение обработки;
- получать доступ только к своим документам и анализам.
