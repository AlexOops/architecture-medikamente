# Рекомендации и приоритеты

## Приоритет P0 — сделать до начала активной миграции

| Мера | Зачем нужна | Риск без меры |
|---|---|---|
| Data profiling текущих Excel/PDF/JPG/1С данных | понять качество и состав данных | перенос дублей, ошибок и мусора |
| Единый master patient id | связать записи пациента между системами | дубли пациентов и ошибки в медкартах |
| Классификация PII/MEDICAL/PAYMENT/LEGAL | понимать, что защищать | утечка чувствительных данных |
| IAM/SSO + базовый RBAC | управлять доступом сотрудников | избыточные права |
| API Gateway для внешних интеграций | контролировать лабораторию и партнёров | хрупкие и небезопасные API |
| Audit log для PII/MEDICAL | расследовать доступ к данным | невозможно доказать факт доступа |
| Backup и план восстановления | снизить риск потери данных | простой и потеря данных |
| План поэтапной миграции | снизить риск big bang migration | остановка бизнеса при сбое |

## Приоритет P1 — внедрить в MVP

| Мера | Зачем нужна | Риск без меры |
|---|---|---|
| ABAC для медицинских данных | учитывать врача, филиал, пациента, цель доступа | доступ не по назначению |
| Policy Decision Point | централизовать правила доступа | дублирование проверок в сервисах |
| DLP для файлов и почты | снизить риск утечек | выгрузки и пересылка PII |
| SIEM и алерты | видеть инциденты | позднее обнаружение утечки |
| Очереди для критичных событий | развязать сервисы | рассинхронизация и потери событий |
| Idempotency keys | безопасно повторять запросы | дубли оплат и записей |
| Централизованные логи и метрики | поддерживать эксплуатацию | долгое расследование сбоев |
| Нагрузочное тестирование MVP | проверить рост нагрузки | деградация при запуске |

## Приоритет P2 — развивать после стабилизации MVP

| Мера | Зачем нужна | Риск без меры |
|---|---|---|
| Data Catalog | управлять владельцами и тегами данных | хаос в данных |
| Data Lineage | понимать путь данных до BI/ML/AI | нельзя корректно удалить/обезличить данные |
| Anonymization Service | безопасно использовать данные в аналитике | утечка через BI/ML |
| Retention & Deletion Service | управлять сроками хранения | нарушение требований к хранению |
| PAM для администраторов | контролировать привилегированный доступ | риск злоупотребления админскими правами |
| EDR на рабочих станциях | защитить endpoints | утечка через рабочие ПК |
| Полноценная Kubernetes-платформа | масштабировать сервисы | сложность эксплуатации при росте |
| DR-site | обеспечить отказоустойчивость | длительный простой при аварии |

## Рекомендуемая последовательность

```text
1. Data audit и классификация
2. Целевая доменная модель
3. IAM/RBAC/ABAC
4. API Gateway и интеграционные контракты
5. MVP CRM/Appointment/EMR
6. Миграция данных волнами
7. Observability и нагрузочное тестирование
8. Secure Analytics Layer
9. Расширение на филиалы
```

## Антипаттерны, которых нужно избежать

- Big bang migration.
- Сразу много микросервисов без зрелой эксплуатации.
- Перенос Excel-хаоса в новую БД.
- Прямые интеграции без API Gateway.
- BI/ML на сырых PII/MEDICAL данных.
- Доступ к медицинским данным только через роль без атрибутов.
- Отсутствие rollback-плана.
