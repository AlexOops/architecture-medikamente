# Отчёт: узкие места при миграции

## Контекст

Компания «Медикаменте» переходит от текущего состояния с Excel, файловыми хранилищами, 1С в файловом режиме и локальным сервером Windows Server к целевой архитектуре с порталами, CRM, EMR, платёжным шлюзом, интеграцией с лабораторией и аналитическим слоем.

Цель миграции — поддержать рост филиалов, сотрудников и клиентов, снизить ручной труд и обеспечить защиту PII и медицинских данных.

## Ключевые процессы, затрагиваемые миграцией

| Процесс | Что меняется | Основной риск |
|---|---|---|
| Регистрация пациента | Excel/сканы -> CRM | потеря данных, дубли, некорректная модель PII |
| Запись на приём | Excel-журнал -> Appointment Service | гонки записей, ошибки синхронизации |
| Медицинская карта | PDF/JPG/Excel -> EMR | сложная миграция документов и медицинских записей |
| Анализы | ручные реестры -> API лаборатории | неверный контракт API, передача лишних данных |
| Оплаты | Excel + 1С -> Payment Gateway + 1С | рассинхронизация платежей и статусов |
| Аналитика | Excel/Jupyter -> Data Lake/DWH | плохое качество данных, отсутствие lineage |
| Доступ сотрудников | AD/общие папки -> IAM/RBAC/ABAC | избыточные права и ошибки ролей |
| Инциденты ИБ | ручной контроль -> SIEM/DLP/Audit | отсутствие событий для расследования |

## 1. Инфраструктура

Проблемы:

- один локальный сервер является точкой отказа;
- сервер совмещает AD, DNS, DHCP, LDAP, файловый сервер и Exchange;
- текущая инфраструктура может не выдержать рост нагрузки;
- нет явного разделения production, test и analytics окружений;
- слабая готовность к Kubernetes и микросервисной эксплуатации.

Риски:

- простой всех систем при отказе сервера;
- деградация производительности при росте клиентов;
- сложность масштабирования филиалов;
- сложность восстановления после сбоя.

Рекомендации:

- разделить инфраструктурные роли;
- подготовить отдельные окружения dev/test/prod;
- внедрить резервное копирование и disaster recovery;
- использовать контейнеризацию постепенно;
- начать с модульного монолита или сервисного выделения доменов, а не с резкого перехода на микросервисы.

## 2. Данные и базы данных

Проблемы:

- данные разбросаны по Excel, PDF, JPG, сканам и 1С;
- нет единого идентификатора пациента;
- есть дубли и неполные записи;
- нет классификации PII/MEDICAL/PAYMENT/LEGAL;
- нет lineage и контроля качества данных.

Риски:

- перенос мусорных данных в новую систему;
- ошибки в медицинских картах;
- невозможность корректно удалить или обезличить данные;
- некорректные BI/ML/AI-выводы.

Рекомендации:

- провести data profiling до миграции;
- ввести master patient id;
- очистить и дедуплицировать данные;
- мигрировать данные поэтапно;
- фиксировать lineage и владельцев данных.

## 3. Интеграции

Проблемы:

- 1С работает в файловом режиме;
- ККМ интегрированы через TCP/IP и OLE-компоненту;
- будущая лабораторная интеграция требует API;
- нет API Gateway и единых контрактов;
- старые процессы могут требовать ручной синхронизации.

Риски:

- рассинхронизация платежей, записей и медицинских данных;
- недоступность внешней лаборатории;
- передача лишних PII во внешние системы;
- хрупкость интеграций при росте нагрузки.

Рекомендации:

- все внешние интеграции вести через API Gateway;
- описать data contracts;
- использовать outbox/inbox pattern для критичных событий;
- добавить idempotency keys;
- внедрить очереди для асинхронной интеграции;
- подготовить fallback-процессы.

## 4. Программное обеспечение и бизнес-логика

Проблемы:

- логика сейчас находится в ручных действиях сотрудников;
- правила записи, оплаты и работы с медкартой не формализованы;
- попытка сразу перейти на микросервисы может привести к распределённому монолиту;
- нет чётких границ доменов.

Риски:

- дублирование бизнес-логики в сервисах;
- сложные распределённые транзакции;
- ошибки при записи пациента или оплате;
- высокая стоимость сопровождения.

Рекомендации:

- выделить домены: CRM, EMR, Appointment, Payment, Laboratory, Analytics;
- описать бизнес-процессы до разработки;
- начинать с модульного монолита или ограниченного набора сервисов;
- использовать event-driven подход только для процессов, где нужна асинхронность;
- завести архитектурные решения ADR.

## 5. Безопасность и конфиденциальность

Проблемы:

- нет классификации и тегирования данных;
- нет детального RBAC/ABAC;
- нет audit log;
- нет DLP/SIEM;
- медицинские данные лежат рядом с пользователями;
- нет механизмов retention и удаления.

Риски:

- утечка PII и медицинских данных;
- пациент получает чужие данные в ЛК;
- сотрудник видит данные не по роли;
- невозможно расследовать инцидент.

Рекомендации:

- внедрить IAM/SSO/MFA;
- добавить RBAC/ABAC;
- использовать Policy Decision Point;
- включить audit log для PII/MEDICAL;
- внедрить DLP и SIEM;
- шифровать данные at rest и in transit;
- не отдавать сырые PII/MEDICAL в BI/ML/AI.

## 6. Персонал и процессы

Проблемы:

- сотрудники привыкли к Excel и ручным операциям;
- IT-команда маленькая;
- нет выделенных специалистов по ИБ, DevOps, Data Engineering;
- бизнес-правила могут быть не документированы.

Риски:

- сопротивление изменениям;
- ошибки при вводе данных;
- недостаток компетенций для эксплуатации микросервисов;
- перегрузка IT-отдела.

Рекомендации:

- назначить владельцев доменов;
- обучить сотрудников новым процессам;
- добавить роли DevOps, Security Engineer, Data Engineer;
- вести миграцию поэтапно;
- оставить временные регламенты rollback и ручного обхода.

## 7. Эксплуатация и наблюдаемость

Проблемы:

- нет централизованных метрик, логов, трассировки;
- нет SLO/SLA;
- нет алертов по бизнес-событиям;
- нет нагрузочного тестирования.

Риски:

- деградация производительности останется незамеченной;
- сложно найти причину сбоя;
- рост филиалов приведёт к нестабильности;
- новые сервисы будут сложно сопровождать.

Рекомендации:

- внедрить централизованные логи;
- добавить метрики и distributed tracing;
- настроить алерты;
- провести нагрузочное тестирование;
- определить SLO для записи, ЛК пациента, EMR, платежей и лаборатории.

## Итог

Основные узкие места миграции находятся в данных, интеграциях, безопасности и операционной зрелости.

Перед активной разработкой необходимо стабилизировать фундамент:

1. классифицировать данные;
2. описать домены и процессы;
3. подготовить IAM/RBAC/ABAC;
4. спроектировать интеграционный слой;
5. провести data profiling;
6. внедрить observability;
7. мигрировать данные поэтапно.
